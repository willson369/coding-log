 # Claude Code 源码泄露事件学习笔记

## 事件概述

- 泄露原因：Anthropic 工程师在发布 Claude Code 2.1.88 版本时，打包工具 Bun 默认生成的 Source Map 文件（cli.js.map）被包含在 npm 发布包中
  > 就像你寄东西给别人，不小心把「如何组装」的说明书也塞进去了，别人收到就能原样复制一份。

- 泄露内容：包含 1906 个 TypeScript 源文件，约 51.2 万行代码
  > 相当于把整个 Claude Code 软件的「配方」全公开了，包括怎么做、怎么运行的所有细节。

- 发现者：安全研究员 Chaofan Shou 在 X 上最先曝光
  > 最早是网名叫 Chaofan Shou 的安全专家发现，并在社交平台公开。

- 泄露方式：map 文件是一个 JSON，包含 sources（文件路径）和 sourcesContent（完整源码）两个关键数组
  > 这个文件其实是个「对照表」，左边是文件名，右边是代码内容，对应着就能还原出所有源码。

## 架构设计亮点

### 1. Agent 循环引擎

- 核心机制：while(true) 无限循环实现 ReAct（推理+执行）模式
  > 让 AI 像人类一样「想一步，做一步，看结果，再想下一步」。

- 工作流程：
  1. 上下文压缩
  2. 调用大模型获取流式响应
  3. 解析工具调用请求
  4. 执行工具
  5. 结果追加到对话历史
  > 就像 AI 的大脑工作流程：先整理思路，问大模型下一步怎么做，大模型说用什么工具，AI 就去执行，把结果记下来，继续循环。

- 模型调用：通过 services/api/claude.ts 中的 queryModel 函数
  > 这是 AI 和大脑（大模型）沟通的入口。

---

### 2. 工具系统设计

- 40+ 内置工具：通过 getAllBaseTools() 函数注册
  > AI 有很多技能，比如读文件、写文件、搜索网页等，都在这里统一登记。

- fail-closed 原则：工具默认不安全（isConcurrencySafe: false, isReadOnly: false），需显式声明安全属性
  > 假设所有工具都危险，除非开发者明确说「我这个是安全的」。

- 动态加载：工具数量多时，先提供精简清单，按需加载完整定义
  > 就像菜单太长，先给你看菜名，你点哪个再给详细做法。

- 并发控制：
  - 默认最大 10 个并发工具
  - 读写分离：只读工具可并发，写操作需排队
  - 使用「读读并行、读写互斥」策略
  > 很多人可以同时看书，但有人要写字，就得等别人看完，不然会乱套。

---

### 3. 三层记忆架构

1. MEMORY.md（热数据）
   - 常驻加载，200 行/25KB 限制
   - 截断时在换行符处切分，避免破坏结构
   - 添加 WARNING 提示
   > 就像随身带的小笔记本，只能记几页，记不下就提醒「后面内容省略」。

2. 话题文件（温数据）
   - 按需加载（最多 5 个相关文件）
   - 用 Sonnet 小模型选择相关记忆
   - 不记录代码（避免缓存不一致）
   > 像书柜里的书，聊到什么拿什么，但不记具体代码，因为代码会变。

3. 历史对话（冷数据）
   - 存储为 JSONL 格式
   - 通过 Grep 搜索检索
   > 旧聊天记录存起来，需要时用搜索查找。

---

### 4. 五级上下文压缩

1. Snip 剪裁：旧工具结果只保留结构
2. Microcompact 微压缩：大体积结果卸载到缓存
3. Context Collapse 折叠：中间对话折叠摘要
4. Autocompact 自动压缩：超过阈值时触发
5. Reactive Compact 应急压缩：API 返回 413 错误时触发

> 就像手机空间不够时的清理策略：先删小文件，再删大文件，再合并重复内容，实在不行紧急清理。

- 断路器机制：连续失败 3 次后停止尝试
  > 像保险丝，连续失败 3 次就自动断电，避免浪费钱和资源。

---

### 5. 系统提示词缓存

- 通过 DYNAMIC_BOUNDARY 标记分割提示词
- 静态部分：全球用户共享缓存
- 动态部分：个性化内容独立加载
- 优化：显著节省 token 成本

> 就像发通知，固定部分（公司抬头）大家共用，动态部分（姓名）单独填，省钱又省事。

---

### 6. 安全设计

- YOLO 模式审查：即使开启危险模式，仍有影子 AI 分类器把关
  > 你说「随便搞」，但背后有个 AI 保镖看着，真危险了还会拦着。

- Bash 安全检测：23 种安全检查规则
  > 有 23 种方法检查你让 AI 执行的命令是否危险。

- 多层权限过滤：
  1. 运行模式检查
  2. 用户自定义钩子
  3. YOLO 分类器
  4. Bash 危险命令拦截
  5. 配置规则引擎
  > 过 5 道安检门，任何一道报警都不让过。

- 反蒸馏机制：向 API 请求注入假工具定义
  > 给偷学的人喂假答案，让他们学歪。

- 卧底模式：内部贡献开源项目时自动隐藏 AI 痕迹
  > 内部员工用 AI 写开源代码时，自动隐藏是 AI 写的，不露马脚。

## 工程技术细节

### 1. 性能优化

- 懒加载：入口文件 cli.tsx 中动态导入模块
  > 不一次全加载，用到哪个功能再加载哪个。

- 早期输入捕获：初始化期间缓存用户输入
  > 启动时你手快打的内容，启动完马上给你用上。

- TCP 预连接：启动时预连接 API，节省 100-200ms 握手时间
  > 一边启动软件，一边提前和服务器「打招呼」，省时间。

---

### 2. Feature Flag 管理

- 实验性功能通过条件 require 引入
- 支持按用户/环境灰度发布
- 未开启功能会被 tree-shaking 移除

> 功能开关，想给谁用就给谁用，不用的功能直接删掉不占空间。

---

### 3. 隐藏功能曝光

- KAIROS 模式：长期助手 + AutoDream（夜间自动整理记忆）
  > AI 24 小时在线，晚上还会「做梦」整理白天的内容。

- COORDINATOR_MODE：多 Agent 协作（研究→综合→实现→验证四阶段）
  > 一个 AI 指挥几个小弟 AI 分工干活。

- VOICE_MODE：语音交互
  > 能用语音和 AI 对话。

- WEB_BROWSER_TOOL：浏览器操作工具
  > AI 能操作浏览器上网。

- 数字宠物系统：18 种虚拟宠物（鸭子、仙人掌等）
  > 写代码写累了可以撸 AI 宠物。

## 可借鉴的设计思想

1. 简单有效胜于复杂：用 Grep 搜索替代 RAG，模型能力强时更有效
2. 分层管理：不同「温度」的数据用不同策略（常驻/按需/搜索）
3. fail-closed 原则：不确定时默认禁止，提高安全性
4. 渐进式压缩：从轻到重逐级应用压缩策略
5. 边界清晰：工程系统与 AI 模型的职责划分明确

## 工程实践启示

1. 发布流程：生产环境必须排除 .map 文件
2. 缓存设计：共享与个性化内容分离
3. 错误处理：添加断路器防止雪崩
4. 性能优化：预连接、懒加载、输入缓冲结合
5. 安全防御：多层防护 + 主动防御策略
